# Deploying Ballast (no Docker)

Ballast deploys from source with native buildpacks via the [`render.yaml`](../render.yaml)
Blueprint at the repo root — one static site (the dashboard) and two services
(the Node core, the Python AI). No Dockerfiles.

The core service runs the **whole backend in one process** — API, scheduler,
reaper, and an in-process worker (`RUN_WORKER=true`) — so it fits on a free
instance. You can split workers into separate `type: worker` services later.

## One-time deploy (Render Blueprint)

1. Push this repo to GitHub (already done).
2. In Render: **New → Blueprint**, select the repo. Render reads `render.yaml`
   and proposes three services: `ballast-core`, `ballast-ai`, `ballast-web`.
3. Fill in the secrets it asks for (the `sync: false` vars):
   - **`DATABASE_URL`** (on `ballast-core` *and* `ballast-ai`) — your Supabase
     **session pooler** URL (host `aws-0-<region>.pooler.supabase.com`, port
     `5432`, user `postgres.<ref>`). Not the transaction pooler (6543).
   - **`ANTHROPIC_API_KEY`** (on `ballast-ai`, optional) — set it to have Claude
     write advisories; leave blank to use the deterministic template.
4. **Apply**. On first build the core runs migrations and seeds the demo account
   automatically.

That's it. The URLs will be:

| Service | URL | What it is |
| ------- | --- | ---------- |
| `ballast-web` | `https://ballast-web.onrender.com` | the dashboard (open this) |
| `ballast-core` | `https://ballast-core.onrender.com` | API (`/health`, `/api/v1/*`) |
| `ballast-ai` | `https://ballast-ai.onrender.com` | advisory service (`/health`, `/analyze`) |

Sign in with the seeded reviewer account **`demo@ballast.dev` / `ballast-demo`**
(prefilled on the login screen).

## Env vars that matter

| Var | Service | Notes |
| --- | ------- | ----- |
| `DATABASE_URL` | core, ai | Supabase session pooler URL (secret) |
| `RUN_WORKER` | core | `true` → the API process also runs a worker |
| `WORKER_ORG_ID` | core | org the in-process worker serves (the demo org) |
| `CORS_ORIGIN` | core | must equal the dashboard URL |
| `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` | core | auto-generated by Render |
| `VITE_API_URL` | web | full API base, e.g. `https://ballast-core.onrender.com/api/v1` (baked at build) |
| `VITE_USE_MOCKS` | web | `false` → talk to the real core |
| `ANTHROPIC_API_KEY` | ai | optional; enables Claude-written advisories |

## Gotchas

- **Names must line up.** `CORS_ORIGIN` and `VITE_API_URL` hard-code the default
  `onrender.com` URLs. If Render gives a service a different name/URL (e.g. a
  name clash adds a suffix), update those two values to match, or CORS/API calls
  fail.
- **Fresh database?** The blueprint's `WORKER_ORG_ID` is the demo org id from the
  reference database. Against a new database, read the id that `db:seed-demo`
  prints in the core's first build log and set `WORKER_ORG_ID` to it.
- **Free instances spin down** after inactivity; the first request after idle
  cold-starts (a few seconds). Fine for a demo.
- **Sessions expire after 15 min** (access-token TTL; refresh isn't wired in the
  UI yet) — just sign in again.

## Alternatives

The same artifacts deploy anywhere without Docker: the **web** build (`npm run
build` → `web/dist`) is static (Vercel / Netlify / Cloudflare Pages); **core**
is a standard Node service (`npm run start`); **ai** is `uvicorn app.main:app`.
Set the same env vars and it works the same.
